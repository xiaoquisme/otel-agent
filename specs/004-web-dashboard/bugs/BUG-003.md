# Bug Report: SQLite Thread Safety Error with ThreadingHTTPServer

**Type**: Implementation drift
**Severity**: Critical
**Feature**: 004-web-dashboard
**Reported**: 2026-06-27

## Description

After switching to `ThreadingHTTPServer` (BUG-002 fix), the dashboard crashes with:
```
sqlite3.ProgrammingError: SQLite objects created in a thread can only be used in that same thread.
```

The persistent `sqlite3.Connection` is created in the main thread but used in request handler threads.

## Artifact Traceability

### spec.md
- **Affected user story**: US1 — Browse Request Logs (P1)
- **Affected requirements**: FR-002 (display table)
- **Gap identified**: Spec didn't account for thread safety when using multi-threaded server

### plan.md
- **Affected sections**: Decision 3 — "Single persistent sqlite3.Connection"
- **Impact**: Persistent connection is incompatible with `ThreadingHTTPServer`

### tasks.md
- **Affected tasks**: T004 (persistent connection)
- **False completions**: T004 works with single-threaded server but breaks with ThreadingHTTPServer

## Root Cause Analysis

`sqlite3.Connection` objects are thread-bound by default. `ThreadingHTTPServer` spawns a new thread per request. The connection created in the main thread cannot be used in request threads.

## Recommended Fix

Add `check_same_thread=False` to the SQLite connection. WAL mode is safe for concurrent reads from multiple threads — the thread check is overly conservative for read-only queries.
