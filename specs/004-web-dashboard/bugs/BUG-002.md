# Bug Report: SSE Endpoint Blocks Entire Dashboard Server

**Type**: Implementation drift
**Severity**: Critical
**Feature**: 004-web-dashboard
**Reported**: 2026-06-27

## Description

The dashboard only shows new data and stops responding after the first SSE connection. When a browser connects to `/api/events`, the SSE handler enters an infinite loop (`while True: ... time.sleep(1)`) that blocks the entire `HTTPServer`. Since `http.server.HTTPServer` is single-threaded, all other requests (HTML page, API data, detail view) are blocked until the SSE connection closes.

**Symptoms:**
- Dashboard loads initially but then freezes
- API requests time out after SSE connection is established
- Only new data appears (via SSE), historical data never loads
- Browser shows blank page or loading spinner

## Artifact Traceability

### spec.md
- **Affected user story**: US1 — Browse Request Logs (P1)
- **Affected requirements**: FR-003 (auto-refresh), FR-002 (display table)
- **Gap identified**: Spec requires auto-refresh via SSE but doesn't address concurrency. The single-threaded HTTP server cannot handle both SSE streaming and regular API requests simultaneously.

### plan.md
- **Affected sections**: Research Decision 2 — "SSE for push updates"
- **Impact**: Plan chose SSE but didn't account for `HTTPServer` being single-threaded. Need to use `ThreadingHTTPServer` or move SSE to a separate thread.

### tasks.md
- **Affected tasks**: T010 (SSE endpoint), T011 (SSE JavaScript)
- **False completions**: T010 marked complete but SSE blocks all other requests
- **Missing tasks**: Need a task to make the server multi-threaded

## Root Cause Analysis

The `HTTPServer` class from `http.server` is single-threaded — it processes one request at a time. The SSE endpoint (`/api/events`) enters an infinite loop that never returns, blocking all subsequent requests. The fix is to use `ThreadingHTTPServer` which spawns a thread per request.

## Recommended Fix

1. Change `HTTPServer` to `ThreadingHTTPServer` in `src/otel_agent/dashboard/server.py`
2. Add a task to verify concurrent request handling
3. Run `/speckit.bugfix.patch` to update spec artifacts
4. Run `/speckit.implement` to apply the fix
