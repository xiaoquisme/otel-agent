# Bug Report: API connection fails with generic error after retries

**Type**: Untested flow
**Severity**: High
**Feature**: 007-multi-provider-routing
**Reported**: 2026-06-30
**Status**: Patched
**Patched**: 2026-06-30

## Description

When attempting to connect to a custom provider endpoint at `http://localhost:8080/openai/v1`, the system retries the connection 3 times and then fails with a generic error message. The error output shows:

```
⚠️  API call failed (attempt 3/3): APIConnectionError
   🔌 Provider: custom  Model: mimo-v2.5-pro
   🌐 Endpoint: http://localhost:8080/openai/v1
   📝 Error: Connection error.
   ⏱️  Elapsed: 21.60s  Context: 2 msgs, ~5,575 tokens
❌ API failed after 3 retries — Connection error.
   💀 Final error: Connection error.
```

The error message "Connection error." is not actionable — it doesn't indicate whether the issue is:
- The endpoint URL is incorrect
- The server is not running
- Network connectivity issues
- Port is blocked or firewall issue
- DNS resolution failure

## Steps to Reproduce

1. Configure a custom provider with `base_url: http://localhost:8080/openai/v1`
2. Ensure the server at that endpoint is not running or unreachable
3. Send a request to the proxy
4. Observe the generic error message after 3 retries

## Root Cause Analysis

The error handling code catches `APIConnectionError` but only surfaces the exception's string representation, which is the generic "Connection error." message from the OpenAI client library. The system doesn't:

1. Inspect the exception for more specific details (e.g., `ConnectionRefusedError`, `TimeoutError`, DNS errors)
2. Add context about what to check (server running? URL correct? firewall?)
3. Distinguish between different failure modes (connection refused vs timeout vs DNS)

This is an untested flow — the acceptance scenario in the spec says errors should be "actionable" but the implementation doesn't meet that bar.

## Recommended Fix

1. Enhance the error handling in `src/otel_agent/addon.py` to:
   - Catch specific exception types (ConnectionRefusedError, TimeoutError, etc.)
   - Add actionable guidance based on the failure type
   - Include the provider name and endpoint in all error messages
   
2. Update error message format to include:
   - What was attempted (provider name, endpoint URL)
   - What failed (specific error type)
   - What to check (server running? URL correct? network connectivity?)

3. Run `/speckit.bugfix.patch` to update spec artifacts (tasks.md) with a new task for enhanced error messages
