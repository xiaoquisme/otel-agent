# Bug Report: Streaming Preview Parser Drops Incomplete Trailing Chunks

**Type**: Implementation drift
**Severity**: High
**Feature**: 018-fix-body-rendering
**Reported**: 2026-07-10
**Status**: Patched
**Patched**: 2026-07-10

## Description

The `parseStreamingChunks()` function in `index.html` silently drops the last JSON chunk when it is truncated (missing closing `}`). Since SSE streaming chunks are concatenated without separators, the last chunk in the preview often has its closing `}` cut off by the storage limit. This causes:

1. The last chunk (often containing the actual content or reasoning) is never parsed
2. `extractStreamingContent()` sees no content deltas → sets `incomplete: true`
3. User sees "Streaming preview may be incomplete" even when the preview contains real data

**Reproduction**: Data ID 864 — preview contains 2 chunks, but parser only finds 1. Second chunk has `reasoning_content: "当前"` but is truncated (missing `}`), so it's dropped.

## Artifact Traceability

### spec.md
- **Affected user story**: US2 — Streaming Response Shows Reassembled Content
- **Affected requirements**: FR-006 (render streaming preview as reassembled markdown), FR-007 (show incomplete indicator when preview too short)
- **Gap identified**: Spec assumed chunks would be complete JSON objects. Did not account for truncation cutting the last chunk mid-JSON.

### plan.md
- **Affected sections**: Phase 0 Research — "Streaming Preview Truncation" section assumed `[:5_000]` would capture complete chunks
- **Impact**: The limit increase helps but doesn't solve the fundamental parsing issue — even at 5000 chars, the last chunk can be truncated

### tasks.md
- **Affected tasks**: T009 (extractStreamingContent) — the `incomplete` flag logic is correct but triggers on truncated data, not missing data
- **False completions**: None — the implementation was correct per the plan, but the plan didn't account for this edge case
- **Missing tasks**: T009 needs to handle the case where chunks are truncated at the preview boundary

## Root Cause Analysis

The `parseStreamingChunks()` function tracks brace depth to find JSON object boundaries. When the preview string ends mid-object (depth > 0), the incomplete chunk is silently ignored. The function returns only complete objects, losing data from the trailing truncated chunk.

This is an **implementation drift** — the code behaves as written but doesn't handle a real-world edge case that the spec didn't anticipate.

## Recommended Fix

1. ✅ Already fixed: `parseStreamingChunks()` now handles incomplete trailing chunks by attempting to close missing braces
2. Run `/speckit-bugfix-verify` to confirm consistency
3. No spec changes needed — the fix is within the existing requirements
