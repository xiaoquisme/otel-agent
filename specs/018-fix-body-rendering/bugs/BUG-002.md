# Bug Report: Truncated Request Body Shows Raw Text Instead of Partial LLM Chat

**Type**: Spec gap
**Severity**: Medium
**Feature**: 018-fix-body-rendering
**Reported**: 2026-07-10
**Status**: Patched
**Patched**: 2026-07-10

## Description

For request ID 1050, the request body exceeds 500KB and gets truncated by the proxy (`server.py:383`). The dashboard correctly detects the truncation and shows the "Body truncated (original exceeded 500KB)" indicator, but falls back to raw text view immediately without attempting to extract and render partial LLM chat messages from the truncated JSON.

Per the spec (FR-004), the system MUST attempt to render truncated request bodies as LLM chat by extracting whatever complete message objects exist in the partial JSON. The current implementation only satisfies FR-003 (detect truncation) and FR-005 (show indicator) but skips FR-004 entirely.

**Reproduction**: Open dashboard, find request ID 1050 (body > 500KB). The body renders as raw text with a yellow truncation banner, rather than showing whatever complete `{"role": "...", "content": "..."}` message objects could be extracted from the partial JSON.

## Root Cause Analysis

This is a **spec gap** — FR-004 was written into the specification but no task was created to implement it. The implementation plan (plan.md) and task list (tasks.md) focus on truncation detection (FR-003) and indicator display (FR-005), but the more complex work of partial JSON parsing and message extraction (FR-004) was never scoped into a task.

The code path in `formatBody()` (`index.html:557-563`) short-circuits on `!isValidJson` — it checks for truncation and shows the indicator, then returns the raw view. There is no attempt to extract partial message objects from the truncated string before falling back.

## Recommended Fix

1. Run `/speckit-bugfix-patch` to add a new task implementing FR-004 (partial LLM rendering from truncated JSON)
2. Run `/speckit-bugfix-verify` to confirm consistency
3. Resume `/speckit-implement` to apply the code fix

**Implementation approach**: In `formatBody()`, after detecting truncation but before returning raw view, attempt to extract complete `{"role": "...", "content": "..."}` objects from the truncated string using a regex or brace-depth parser. If any complete messages are found, render them as formatted LLM chat above the raw view. If none are found, fall back to the current raw view with indicator.
