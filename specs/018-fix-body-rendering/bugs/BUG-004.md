## Description

After T018 fix (renderChatMessage routes role=tool through highlightJsonString), tool message content for data ID 1098 still does not render correctly in the formatted chat view. The user reports seeing raw/unformatted content instead of syntax-highlighted JSON.

The tool messages contain structured JSON responses (search results, file reads, terminal output) with fields like `matches_text` containing multi-line code snippets with embedded newlines. These should display as formatted, syntax-highlighted JSON in the chat view.

**Reproduction**: Open dashboard, find request ID 1098 (99 messages, tool-heavy conversation). Click to view request body in formatted mode. Tool messages should show syntax-highlighted JSON but appear unformatted.

## Root Cause Analysis

T018 added `role === 'tool'` detection in `renderChatMessage` that routes through `highlightJsonString`. However:

1. `extractOpenAiRequestContent` already transforms tool content via `JSON.stringify(JSON.parse(content), null, 2)`. If `JSON.parse` fails on the original content (e.g., truncated or malformed), the raw string is passed through. Then `renderChatMessage` tries `JSON.parse` again — if it also fails, falls back to `escapeHtml` which renders unformatted text.

2. For tool messages where the original content IS valid JSON but very large (3000+ chars with deeply nested objects), `highlightJsonString` produces a massive HTML tree that may be slow to render or visually overwhelming.

3. Multi-line string values (like `matches_text` with embedded `\n` characters) render correctly in `<pre>` with `white-space: pre-wrap`, but the long lines may overflow horizontally.
