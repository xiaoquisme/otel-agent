# Bug Report: DuckDB Lock Conflict Between Proxy and Dashboard

**Status**: Patched
**Patched**: 2026-07-09

**Type**: Spec gap
**Severity**: Critical
**Feature**: 013-sqlite-to-duckdb-migration
**Reported**: 2026-07-09

## Description

When the proxy (TelemetryLogger) is running and writing to the DuckDB database, the dashboard's SSE endpoint crashes with an `IOException` when trying to open a read-only connection to the same file.

```
_duckdb.IOException: IO Error: Could not set lock on file
"/Users/xiaoqu/.otel-agent/telemetry.duckdb": Conflicting lock is held
in /Users/xiaoqu/anaconda3/bin/python3.13 (PID 90346) by user xiaoqu.
```

The proxy process holds an exclusive lock on the `.duckdb` file. When the dashboard (a separate process) tries to open it with `read_only=True`, DuckDB still requires a file lock and fails because the exclusive lock is already held.

This is a fundamental DuckDB limitation: **DuckDB does not support multi-process concurrent access to the same database file**. Unlike SQLite with WAL mode (which allows one writer + multiple readers across processes), DuckDB's file lock is exclusive — only one process can hold any connection at a time.

## Artifact Traceability

### spec.md
- **Affected requirement**: FR-010 — "System MUST handle concurrent reads (dashboard) and writes (proxy logging) without deadlocks or data corruption."
- **Affected assumption**: "DuckDB supports WAL-like concurrent read/write access for the proxy+dashboard use case."
- **Gap identified**: The spec assumes DuckDB has the same multi-process concurrency model as SQLite WAL mode. This is incorrect — DuckDB uses an exclusive file lock per database file. FR-010 is impossible to satisfy with the current single-file architecture.

- **Impact**: This is misleading. DuckDB's MVCC provides in-process concurrency (multiple threads in the same process), but NOT multi-process concurrency. The research doc should have noted this critical distinction and proposed a solution.

## Root Cause

The root cause is a spec-level oversight: the migration assumed DuckDB's concurrency model matches SQLite's WAL mode. The research document's comparison table listed "Concurrent reads: Automatic" without qualifying that this only applies to in-process (multi-threaded) access, not multi-process access.

In the current architecture, the proxy and dashboard run as separate OS processes (the proxy is a long-running HTTP server; the dashboard is a separate HTTP server started by `otel-agent dashboard`). Each opens its own connection to the `.duckdb` file. DuckDB prevents this by holding an exclusive file lock.

SQLite with WAL mode solved this naturally: one writer process + multiple reader processes could all access the same `.db` file simultaneously.

## Recommended Fix

Multiple approaches, ordered by recommended priority:

**Option A (Recommended): Single-process architecture** — Run the dashboard's database reads through the proxy process via an internal API or shared connection pool. The proxy already holds the DuckDB connection; the dashboard could call the proxy's API for data instead of opening its own connection. This keeps DuckDB's single-process model intact.

**Option B: SQLite fallback for dashboard** — If the dashboard must be a separate process, keep SQLite as the storage backend and abandon the DuckDB migration for the concurrent access use case. DuckDB would only be used for CLI analysis (offline queries on a copy).

**Option C: DuckDB httpfs / API server** — Use DuckDB's ability to query remote/HTTP-based databases. Not practical for the `.duckdb` file use case.
