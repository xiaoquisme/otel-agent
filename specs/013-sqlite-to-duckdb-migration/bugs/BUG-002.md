# Bug Report: Dashboard Health Check Timeout Causes Intermittent DuckDB Lock Conflict

**Type**: Implementation drift
**Severity**: High
**Feature**: 013-sqlite-to-duckdb-migration
**Reported**: 2026-07-09

**Status**: Patched
**Patched**: 2026-07-09

## Description

After the BUG-001 fix was deployed (dashboard routes queries through proxy's internal API), the DuckDB lock conflict still occurs intermittently. The root cause is that `DashboardAPI._proxy_url()` performs a live HTTP health check to the proxy on EVERY query with a 2-second timeout. When the proxy is busy handling requests, the health check times out, causing `DashboardAPI` to fall back to opening a direct DuckDB connection — which conflicts with the proxy's exclusive lock.

**Reproduction**:
1. Start proxy (holds DuckDB lock)
2. Start dashboard (detects proxy, sets proxy_port)
3. Send several requests through the proxy (proxy is busy)
4. Dashboard queries arrive while proxy is processing → health check times out → fallback to direct DuckDB → IOException

**Error**:
```

"/Users/xiaoqu/.otel-agent/telemetry.duckdb": Conflicting lock is held
in /Users/xiaoqu/anaconda3/bin/python3.13 (PID 90346) by user xiaoqu.
```

## Root Cause

The `DashboardAPI._proxy_url()` method (in `api.py`) does a live HTTP health check on every single query:

```python
def _proxy_url(self) -> str | None:
    if self._proxy_port is None:
        return None
    url = f"http://127.0.0.1:{self._proxy_port}"
    try:
        req = urllib.request.Request(f"{url}/health", method="GET")
        with urllib.request.urlopen(req, timeout=2) as resp:
            if resp.status == 200:
                return url
    except (urllib.error.URLError, OSError, TimeoutError):
        pass
    return None  # Falls back to direct DuckDB!
```

When the proxy is busy (handling LLM requests with 300s timeout), the2-second health check can fail, causing `DashboardAPI` to open a direct DuckDB connection — which conflicts with the proxy's exclusive lock.

## Recommended Fix

1. Cache the proxy URL after first successful check (TTL-based, e.g. 30s)
2. Never fall back to direct DuckDB if proxy was previously reachable
3. Only retry direct DuckDB if proxy has been unreachable for > 60s

Specifically:
```python
def _proxy_url(self) -> str | None:
    if self._proxy_port is None:
        return None
    now = time.monotonic()
    # Return cached result if recent
    if self._proxy_url_cache is not None and now - self._proxy_url_cache_time < 30:
        return self._proxy_url_cache
    url = f"http://127.0.0.1:{self._proxy_port}"
    try:
        req = urllib.request.Request(f"{url}/health", method="GET")
        with urllib.request.urlopen(req, timeout=2) as resp:
            if resp.status == 200:
                self._proxy_url_cache = url
                self._proxy_url_cache_time = now
                return url
    except (urllib.error.URLError, OSError, TimeoutError):
        pass
    # If proxy was previously reachable, keep using it (avoid lock conflict)
    if self._proxy_url_cache is not None:
        return self._proxy_url_cache
    return None
```
