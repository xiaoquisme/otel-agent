Ideation: Frontend-Backend Separation for otel-agent Dashboard

2026-07-13 Topic: frontend-backend-separation Focus: LLM response rendering framework + dashboard decoupling Mode: repo-grounded

Grounding Context

Project: otel-agent — LLM API gateway with OpenAI/Anthropic compatible endpoints, model-name-based provider routing, and telemetry logging.

Stack: Python 3.10+, FastAPI (proxy), httpx, DuckDB/SQLite, PyYAML. Build: hatchling + uv.

Dashboard architecture: dashboard/server.py (BaseHTTPRequestHandler, 226 lines) serves a monolithic index.html (995 lines, ~500 lines JS) + JSON API endpoints. dashboard/api.py reads from DuckDB via proxy internal API or direct connection.

Core pain: ~500 lines of hand-rolled JS handle LLM response format detection (OpenAI/Anthropic/streaming), content extraction, markdown rendering (marked.js + DOMPurify), tool call display, JSON syntax highlighting, and streaming chunk parsing. This duplicates format knowledge already in converter.py. Bug specs 011, 012, 018 (4 bug reports), 020 are all dashboard rendering issues.

Historical decisions: Spec 004 mandated "no JS frameworks — plain HTML/CSS/JS." Spec 016 said "~800 lines is acceptable." The file is now 995 lines and growing. DuckDB lock constraint requires dashboard queries to route through proxy's internal HTTP API.

Topic Axes

  1. Dashboard UI rendering — LLM response display, markdown, streaming, tool calls, format detection
  2. Backend-for-frontend API — HTTP handler, SSE, JSON endpoints, data flow
  3. Frontend architecture — single-file HTML vs SPA vs framework, component structure
  4. Development workflow — testing, iterating, separating frontend/backend
  5. Deployment & packaging — how dashboard is served, embedded vs standalone

Ranked Ideas

Ideas

  1. Merge Dashboard into FastAPI Proxy
  2. Backend-Side LLM Format Renderer + Normalization API
  3. React + Vite + Vercel AI SDK Frontend
  4. Provider Format Plugin Registry
  5. Python Test Harness for LLM Body Rendering
  6. ES Modules + Alpine.js Component System

1. Merge Dashboard into FastAPI Proxy

Deployment & packaging Confidence: High Complexity: Medium

Basis: direct: — DashboardAPI._proxy_url() is 45 lines of health-check caching that exists solely because two servers share one DuckDB file. dashboard/server.py manually reimplements URL routing (do_GET with urlparse/parse_qs) that FastAPI does declaratively. BUG-001/BUG-02 workarounds exist only because of the two-process architecture.

Why it matters: Eliminates the entire class of multi-process DuckDB bugs (BUG-001, BUG-002). Removes ~226 lines of legacy BaseHTTPRequestHandler code. Unifies deployment to one process, one port. Gives the dashboard access to FastAPI middleware (CORS, auth, logging, async) for free. Every future feature (WebSocket upgrades, rate limiting) no longer needs dual implementation.

Downsides: Requires migrating all 7 dashboard endpoints to FastAPI routes. SSE implementation changes from manual threading to StreamingResponse. Needs careful handling of the static file serving for the HTML dashboard.

Meeting test: "How many HTTP servers are running?" Today: 2. After: 1.

2. Backend-Side LLM Format Renderer + Normalization API

Backend-for-frontend API Confidence: High Complexity: Medium

Basis: direct: — detectFormat() (index.html:503-511) duplicates format knowledge in converter.py. The four extract*Content() functions (index.html:596-654) reimplement what Python already does. Adding a render_body parameter to the dashboard API that returns pre-rendered HTML is a natural extension of the existing DashboardAPI class.

Why it matters: This is the highest-impact idea for the user's stated pain. The ~500 lines of JS format detection, extraction, and rendering collapse to ~50 lines of HTML embedding. Format changes (adding Google Gemini, Mistral) need only one Python change instead of parallel JS updates. Rendering becomes testable with pytest instead of browser automation. The converter.py already has format expertise — this unifies it.

Downsides: Pre-rendered HTML is format-locked at render time. Server-side rendering adds latency to detail panel opens. Streaming previews still need special handling (pre-render at storage time vs. at request time).

Meeting test: "If we add a new provider format, do we need to update the frontend JS?" — Answer should be "no."

3. React + Vite + Vercel AI SDK Frontend

Frontend architecture Confidence: High Complexity: High

Basis: external: — Vercel AI SDK (ai npm package) is the gold standard for LLM UI rendering. It handles OpenAI/Anthropic format detection, streaming SSE, tool call assembly, and reasoning token display natively. @ai-sdk/openai and @ai-sdk/anthropic providers handle streaming SSE parsing. Paired with react-markdown + rehype-highlight for rich rendering. reasoned: — React+Vite is the ecosystem where Vercel AI SDK is native; ~45KB gzipped bundle is acceptable for a developer tool.

Why it matters: This is the "use an existing rendering framework" path the user explicitly asked about. Vercel AI SDK eliminates: manual detectFormat(), manual parseStreamingChunks(), manual extract*Content() functions, manual highlightJsonString(), and manual renderMarkdown(). The SDK's useChat hooks handle streaming state, message assembly, and tool call rendering. Combined with @tanstack/react-table for the request list and recharts for charts, this replaces the entire 995-line monolith with a proper component architecture.

Downsides: Highest migration effort. Introduces npm/node toolchain. Changes deployment model (built assets served by FastAPI). Breaks the "no JS frameworks" precedent from Spec 004. ~45KB gzipped runtime overhead.

Meeting test: "Does the user want to use a rendering framework they don't have to maintain?" — If yes, this is the answer.

4. Provider Format Plugin Registry

Frontend architecture Confidence: Medium Complexity: Medium

Basis: reasoned: — Currently adding a new LLM format requires editing converter.py (4 conversion functions), adding format detection branches in 5+ JS functions in index.html, updating AUTH_HEADERS in server.py, and adding tests in test_converter.py. A plugin registry where each format bundles its own converter, renderer, auth headers, and tests makes each format a single file. direct: — The AUTH_HEADERS dict in server.py and detectFormat() in JS are manual registries that must be updated per-format.

Why it matters: Compounds because every subsequent provider addition becomes trivial. Without it, each new format multiplies the surface area across 3+ subsystems. With it, the marginal cost of supporting a new format approaches zero. This is the strongest second-order compounding choice in the set.

Downsides: Requires upfront design of the plugin interface. May be premature optimization if only 2-3 providers are planned. Adds indirection that new contributors must understand.

Meeting test: "What's the checklist for adding Google Gemini support?" Today: 5+ files, 3+ subsystems. With plugin registry: 1 new file.

5. Python Test Harness for LLM Body Rendering

Development workflow Confidence: High Complexity: Low

Basis: direct: — Spec 018 had 4 separate bug reports (BUG-001 through BUG-004) for body rendering alone, all discovered after deployment. The detectFormat() function (index.html:503-511) has 6 format branches with edge cases — zero Python test coverage. Extracting format detection and rendering into a testable Python module (dashboard/render.py) with 30+ fixture bodies would catch these before they reach users.

Why it matters: Every dashboard spec from 011 onward has had post-deployment bugs in body rendering. This is the highest-ROI change because it addresses the root cause (untested format logic) without requiring a full architecture rewrite. It also unblocks Ideas 2 and 3 by creating the testable Python renderer that those ideas depend on.

Downsides: Doesn't solve the 500-line JS problem by itself — the frontend still has its own rendering code. Is a stepping stone, not a destination.

Meeting test: "How many test cases cover the format detection function?" Currently zero in Python; should be 12+.

6. ES Modules + Alpine.js Component System

Frontend architecture Confidence: Medium Complexity: Low

Basis: direct: — The 995-line index.html contains CSS (~170 lines), HTML (~100 lines), and JS (~720 lines) in one file with no module system. The LLM rendering functions (lines 500-820) form a coherent API boundary that could be an ES module. external: — Alpine.js (~15KB) provides x-data, x-for, x-show reactivity without a build step. ES modules work natively in modern browsers via <script type="module">.

Why it matters: This is the minimal-disruption path toward componentization. It preserves the "no build tool" constraint from Spec 004 while gaining module isolation. Each LLM format (OpenAI, Anthropic, streaming) becomes its own module. The rendering framework choice (marked.js, react-markdown, or a custom renderer) slots in per-component. This could be a stepping stone to Idea 3 (React) or a permanent lightweight solution.

Downsides: Doesn't fully solve the LLM format detection problem — the JS still needs to detect and render formats. Alpine.js has a smaller ecosystem than React. No TypeScript type safety without a build step.

Meeting test: "If I change the markdown rendering logic, do I have to look at the table pagination code?" Currently yes; after split, no.
