Grounding Context

Project: otel-agent — FastAPI LLM API gateway with model-name-based routing.

Current state: Client must specify exact model (e.g. openai/gpt-4o). No auto mode, no task detection, no cost awareness, no fallback. Routing is pure prefix-based: parse_model() splits on /, resolve_provider() looks up the provider. Provider dataclass has 4 fields: name, base_url, api_key, api_format — no costs, no tiers, no capabilities.

External patterns: LiteLLM auto_router (heuristic/LLM/keyword classification, tier system SIMPLE→REASONING, Thompson sampling, fallback chains). OpenRouter auto (NotDiamond ML routing, session stickiness, model aliases).

Key files: router.py (40 lines), config.py (154 lines), server.py (441 lines), models.py (112 lines), logger.py (captures latency/usage per request).

1. Auto Mode Entry Point

Axis: routing-strategy
Confidence: High (90%)
Complexity: Low
Description:

Intercept model="auto" before parse_model() in server.py. When the client sends "auto", route through a task classifier instead of erroring out. Explicit model strings (openai/gpt-4o) continue to work as before — "auto" is the special case, not a replacement.

Basis:

direct: parse_model() in router.py raises ValueError on any input without a slash. Every client must hardcode provider prefix — there's no graceful degradation.

Rationale:

This is the literal feature gate. Without it, nothing else exists. It's also the highest-friction entry point for new users.

Downsides:

Requires a classifier to be useful (idea #2). Without one, "auto" is just a default model picker.

2. Heuristic Task Classifier

Axis: task-classification
Confidence: High (85%)
Complexity: Low
Description:

A zero-cost heuristic that classifies incoming requests into complexity tiers (SIMPLE / MEDIUM / COMPLEX / REASONING) using structural signals: token count estimate, presence of code blocks, tool/function definitions, message chain length, reasoning keywords. Runs in <1ms with no LLM call. Placed in a new classifier.py module.

Basis:

external: LiteLLM's auto_router uses heuristic classification as the default strategy — zero API calls, sub-millisecond. The specific signals (code blocks = code gen → needs larger model; short single-turn = simple → cheap model) are well-established.

Rationale:

Without tiering, every request looks identical. You can't optimize cost or quality because you don't know if a request needs GPT-4-class reasoning or could be handled by a $0.01/1K token model.

Downsides:

Heuristic accuracy is ~80% — some requests will be misclassified. A fallback mechanism (idea #5) mitigates this.

3. Provider Capability Registry

Axis: configuration-schema
Confidence: High (90%)
Complexity: Low
Description:

Extend the Provider dataclass with routing metadata: cost_per_1k_input, cost_per_1k_output, max_context, rate_limit_rpm, and a tiers field mapping each model to SIMPLE/MEDIUM/COMPLEX/REASONING. Config becomes the single source of truth for routing decisions.

Basis:

direct: Current Provider dataclass (config.py:47) has zero routing-relevant fields. LiteLLM's model_prices_and_context_window.json is a massive external file that should be user-declarative instead.

Rationale:

Without provider capabilities declared in config, any auto-routing must hardcode cost tables (fragile) or call external APIs (slow). This is the prerequisite that makes all other routing ideas possible.

Downsides:

Users must keep cost data current. Could auto-populate from provider APIs as a fallback.

4. Cost-Optimized Routing with Thompson Sampling

Axis: routing-strategy
Confidence: Medium (75%)
Complexity: Medium
Description:

Within each complexity tier, maintain a running score per provider using Thompson Sampling — a Bayesian bandit algorithm. Each request samples a provider based on historical success/failure rates, weighted by declared cost. Cheaper + reliable providers get sampled more. New/cheap providers explored occasionally. Replaces static cost tables with adaptive, observed behavior.

Basis:

external: LiteLLM uses Thompson sampling for pool selection within tiers. The algorithm is well-understood (Netflix thumbnails, Microsoft ad serving). Each "pull" is a request that succeeds or fails — binary reward with continuous cost dimension.

Rationale:

Static cost tables become stale the day a provider changes pricing or a new model drops. Thompson Sampling adapts automatically — if xiaomi/mimo-v2.5 is cheap and reliable, the router discovers this organically.

Downsides:

Requires idea #2 (classifier) and #3 (capability registry) as prerequisites. Cold start is noisy — first ~100 requests per tier are exploratory.

5. Cascade Fallback Chains with Circuit Breaking

Axis: fallback-resilience
Confidence: High (85%)
Complexity: Medium
Description:

Define ordered fallback chains in config (e.g., auto.primary: [openai/gpt-4o, openrouter/claude-3.5] → auto.fallback: [xiaomi/mimo-v2.5]). On 502/504/rate-limit, retry with next provider. Track failure counts per provider — "open the circuit" after N consecutive failures, half-open probe after cooldown. Battle-tested SMTP MX preference pattern.

Basis:

external: SMTP MX preference chains (RFC 5321) — 40-year-old solved problem. direct: server.py catches ConnectError/TimeoutException but returns immediately — no retry, no fallback.

Rationale:

Provider outages are the #1 reliability issue in multi-provider setups. A cascade chain means zero-downtime for the client even when one provider is down.

Downsides:

Fallback adds latency on provider failures. Circuit breaker thresholds need tuning.

6. Session-Sticky Routing with Tier Escalation

Axis: routing-strategy
Confidence: High (80%)
Complexity: Low
Description:

Detect multi-turn conversations (messages array with 3+ turns or client-provided X-Session-ID header) and pin to same provider for duration. First request goes through auto-routing; subsequent requests bypass classification and use cached decision. If quality signals degrade within tier, escalate to next tier up. Prevents context-model mismatch.

Basis:

external: OpenRouter explicitly supports session stickiness for auto-routing. direct: Current codebase has no session concept — every request independently routed.

Rationale:

Multi-turn is the dominant LLM use case. Routing each turn independently silently degrades quality — different formatting, different knowledge cutoffs, different personality. This is a correctness fix, not just an optimization.

Downsides:

Requires session state (in-memory cache with TTL). Cold start on new sessions always runs full classification.

7. Routing Decision Audit Trail

Axis: observability-feedback
Confidence: High (85%)
Complexity: Low
Description:

When auto-routing selects a provider/model, inject response headers (X-Routed-Provider, X-Routed-Model, X-Routed-Tier, X-Routed-Reason) so the client can see exactly why. Log the same to telemetry database. Like Cloudflare's cf-ray header — full provenance for every decision.

Basis:

direct: Telemetry logger captures everything internally but clients see nothing. Server.py doesn't inject routing metadata into responses. The data exists; only the transparency layer is missing.

Rationale:

The #1 complaint about auto-routing systems is opacity. "Why did it pick GPT-3.5 for my complex task?" Without transparency, you can't debug, can't trust, and can't improve.

Downsides:

Minimal — additive to existing response handling. Slight increase in response header size.
